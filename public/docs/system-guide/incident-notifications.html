Incident Notifications

The incident framework surfaces business-state problems — conditions that exist until someone fixes them — through a bell in the top navigation, a dashboard panel, and a toast on first detection. Incidents are global (they describe the state of the business, not a per-user inbox) and self-clearing: when the underlying condition resolves, the incident closes for everyone.

Architecture

PieceLocationRole
IncidentDetectorapp/Modules/Core/Contracts/Contract each detector implements: type() returns a stable slug; check() returns null (condition absent) or {title, message, metadata} (condition present).
IncidentSyncServiceapp/Modules/Core/Services/Runs every registered detector; opens a NotificationIncident when check() returns data, updates it while the condition persists, resolves it when check() returns null.
NotificationIncidentapp/Modules/Core/Models/Eloquent model with query scopes for active/resolved incidents. One row per detector type per occurrence.
UnpostedInvoicesIncidentDetectorapp/Modules/Purchasing/Services/The first concrete detector (see below).

UI components

  • resources/views/livewire/incident-bell.blade.php — badge + dropdown in the top navigation. Polls with wire:poll.20s="checkIncidents"; the poll is also what triggers IncidentSyncService, so detection runs as long as anyone has a page open.
  • resources/views/livewire/incident-dashboard-panel.blade.php — active-incident panel on the dashboard, same 20-second poll.
  • resources/views/components/incident-toast.blade.php — transient toast when a new incident appears during a session.
Two layouts, both wired The bell is mounted in both live layouts — resources/views/layouts/app.blade.php and resources/views/components/layout/app.blade.php. Any change to how the bell is embedded must be made in both, mirroring the dual-nav convention.

Visibility is permission-gated: users without documents-view-any don't see the bell or panel.

The unposted-invoices detector

UnpostedInvoicesIncidentDetector fires while any purchase invoice sits in received, reviewed, or approved — extracted but not yet at a terminal state. Its metadata carries the total and a per-status breakdown. It clears once every such invoice has been posted or rejected.

Adding a new detector

  1. Create a class implementing IncidentDetector. Return a stable slug from type(); make check() cheap — it runs on every 20-second poll.
  2. Register it in AppServiceProvider, where IncidentSyncService is constructed with its detector array (around the existing UnpostedInvoicesIncidentDetector entry).
  3. No UI changes needed — the bell, panel, and toast render whatever the sync service produces.
  4. Add a feature test asserting the incident opens on the trigger condition and resolves when it clears. Note: assert on metadata values, not array key order.
