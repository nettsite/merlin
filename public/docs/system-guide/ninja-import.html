InvoiceNinja Import

Merlin ships a one-shot migration command for moving billing data out of a self-hosted InvoiceNinja v5 installation. It reads the Ninja install directly and creates the equivalent Merlin records, so an existing billing history carries over instead of starting from zero.

Usage

php artisan ninja:import {path?} [--dry-run] [--only=phase] [--limit=N]
Argument / optionMeaning
pathThe InvoiceNinja install directory (e.g. /srv/ninja). Prompted for if omitted.
--dry-runShow what would be imported — counts only, no writes.
--only=Run a single phase: clients, contacts, invoices, quotes, credits, payments, or recurring.
--limit=NCap rows per phase — useful for a trial run against production data. 0 (default) means no limit.

What gets imported

The import runs in phases, in dependency order. Ninja clients become Merlin parties with client relationships, each getting its own receivable sub-account if a control account is configured; their contacts become persons with contact assignments; invoices, quotes, and credits become Document rows of the matching type; payments attach to their invoices; recurring profiles become recurring invoices. Internal ID maps (Ninja client id → Merlin party UUID, Ninja invoice id → Merlin document UUID) keep cross-references intact across phases.

Discount reconstruction

Ninja applies discounts at two levels that don't show up anywhere in a naive read of line_items:

  • Per-line discount — each line item carries its own discount / is_amount_discount pair, separate from cost and quantity.
  • Document-level discount — invoices.discount / invoices.is_amount_discount (also present on quotes and credits) applies on top of every line combined.

Both are folded into a single DocumentLine.discount_amount per line during import (the model only honours one of discount_amount/discount_percent, so the two sources are resolved to one absolute value rather than stacked as separate fields). An amount-based document discount is prorated across lines by their share of the pre-discount subtotal; a percentage-based one applies uniformly to each line's already-line-discounted amount. Left unhandled, reconstructed totals overstate Ninja's real invoice amount — the invoice ends up with a phantom balance_due even though it's fully paid in Ninja.

"Apply credit" payments

Ninja represents applying a client's credit balance to an invoice as a payment record with amount = 0 on the header — the real value exists only in that payment's paymentables allocation. The payments phase does not use the header amount to decide whether a payment has anything to import; it checks whether any allocation to an invoice is non-zero. The resulting Merlin payment document's total is set to the sum of what was actually applied to invoices (after any overpayment capping — see below), not Ninja's header figure.

Overpayment capping

If a payment's allocation to an invoice exceeds that invoice's remaining balance (rounding in the source data), the amount applied is capped at the balance due rather than rejected. The payment document's total reflects the capped amount actually applied, not the raw allocation — otherwise reports would credit more to accounts receivable than was really settled.

Run dry first Always run with --dry-run first and check the phase counts against what you expect from the Ninja install. Then run phases individually with --only= if you want to verify each step, or run the whole command for a full import.

Command registration

The command lives in app/Modules/Billing/Console/ImportFromNinja.php. Module console commands are not auto-discovered — they're registered explicitly in AppServiceProvider::register(). If the command doesn't appear in php artisan list, check that registration.
